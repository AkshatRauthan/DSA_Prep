# C++ PMR (Polymorphic Memory Resources) Guide

## What is PMR?

`std::pmr` (Polymorphic Memory Resource) lets you plug in a **custom memory pool** into standard containers at runtime — without changing their type. All allocations go through your pool instead of the default heap (`new/delete`), which is significantly faster for competitive programming and performance-critical code.

---

## Setup

```cpp
#include "memory_resource"
using namespace std;

// Global pool — allocates from a shared unsynchronized pool
// unsynchronized = no thread safety overhead (fine for CP)
pmr::unsynchronized_pool_resource pool;
```

`unsynchronized_pool_resource` is the fastest option when you're single-threaded (which is always the case in CP).

---

## Using PMR with Common Containers

### `pmr::vector`

```cpp
pmr::vector<int> v(&pool);
pmr::vector<int> v2(100, 0, &pool);   // size 100, all zeros
```

### `pmr::vector` of `pmr::vector` (2D)

```cpp
pmr::vector<pmr::vector<int>> dp(&pool);

// Push rows
for (int i = 0; i < n; i++)
    dp.push_back(pmr::vector<int>(m, 0, &pool));
```

Or preallocate:
```cpp
pmr::vector<pmr::vector<int>> dp(n, pmr::vector<int>(m, 0, &pool), &pool);
```

---

### `pmr::unordered_map`

```cpp
pmr::unordered_map<char, int> freq(&pool);
freq['a']++;
```

### `pmr::map` (ordered)

```cpp
pmr::map<int, int> m(&pool);
m[1] = 10;
```

---

### `pmr::unordered_set` / `pmr::set`

```cpp
pmr::unordered_set<int> seen(&pool);
seen.insert(5);

pmr::set<int> ordered(&pool);
ordered.insert(3);
```

---

### `pmr::string`

```cpp
pmr::string str("hello", &pool);
str += " world";
```

---

### `pmr::list` (Doubly Linked List)

```cpp
pmr::list<int> lst(&pool);
lst.push_back(10);
lst.push_front(5);
lst.pop_back();
lst.pop_front();

// iteration
for (auto x : lst) cout << x << " ";
```

---

### `pmr::deque`

```cpp
pmr::deque<int> dq(&pool);
dq.push_back(1);
dq.push_front(2);
dq.pop_back();
dq.pop_front();
```

---

### Stack

`std::stack` doesn't support PMR directly, but you can back it with a `pmr::deque`:

```cpp
stack<int, pmr::deque<int>> st(pmr::deque<int>(&pool));

st.push(1);
st.push(2);
st.top();   // 2
st.pop();
```

---

### Queue

Same pattern — back it with a `pmr::deque`:

```cpp
queue<int, pmr::deque<int>> q(pmr::deque<int>(&pool));

q.push(1);
q.push(2);
q.front();  // 1
q.pop();
```

---

### Max Heap (Priority Queue)

`std::priority_queue` doesn't natively support PMR, but you can back it with a `pmr::vector`:

```cpp
// max heap — largest element on top (default)
priority_queue<int, pmr::vector<int>> maxHeap(less<int>(), pmr::vector<int>(&pool));

maxHeap.push(3);
maxHeap.push(1);
maxHeap.push(5);
maxHeap.top();   // 5
maxHeap.pop();
```

### Min Heap

```cpp
// min heap — smallest element on top
priority_queue<int, pmr::vector<int>, greater<int>> minHeap(greater<int>(), pmr::vector<int>(&pool));

minHeap.push(3);
minHeap.push(1);
minHeap.push(5);
minHeap.top();   // 1
minHeap.pop();
```

### Heap of Pairs (common in Dijkstra / LC problems)

```cpp
using T = pair<int,int>;

// max heap by first element
priority_queue<T, pmr::vector<T>> maxPairHeap(less<T>(), pmr::vector<T>(&pool));

// min heap by first element
priority_queue<T, pmr::vector<T>, greater<T>> minPairHeap(greater<T>(), pmr::vector<T>(&pool));

minPairHeap.push({3, 0});
minPairHeap.push({1, 2});
minPairHeap.top();  // {1, 2}
```

## Pool Options

| Pool Type | Use Case |
|---|---|
| `unsynchronized_pool_resource` | Single-threaded, fastest — use this in CP |
| `synchronized_pool_resource` | Multi-threaded, thread-safe |
| `monotonic_buffer_resource` | Fastest possible, but NO deallocation (use for read-only / throwaway allocations) |

### `monotonic_buffer_resource` — Ultra Fast

```cpp
// Fixed buffer on stack — zero heap overhead
char buffer[1 << 20]; // 1MB stack buffer
pmr::monotonic_buffer_resource mono(buffer, sizeof(buffer));

pmr::vector<int> v(&mono);
pmr::unordered_map<int,int> m(&mono);
// No individual frees happen — all released at once when mono goes out of scope
```

Use this when you **know the total memory needed upfront** and don't need to free individual elements.

## Full Example — LC Style

```cpp
#include "iostream"
#include "vector"
#include "unordered_map"
#include "memory_resource"
using namespace std;

pmr::unsynchronized_pool_resource pool;

string minWindow(string s, string t) {
    pmr::unordered_map<char, int> need(&pool), window(&pool);
    for (char c : t) need[c]++;

    int sIdx=0, minLen=INT_MAX, minSIdx=0;
    int have=0, required=need.size();

    for (int i=0; i<s.length(); i++) {
        window[s[i]]++;
        if (need.count(s[i]) && window[s[i]] == need[s[i]])
            have++;

        while (have == required) {
            if (i-sIdx+1 < minLen) { minLen=i-sIdx+1; minSIdx=sIdx; }
            window[s[sIdx]]--;
            if (need.count(s[sIdx]) && window[s[sIdx]] < need[s[sIdx]])
                have--;
            sIdx++;
        }
    }

    return minLen == INT_MAX ? "" : s.substr(minSIdx, minLen);
}
```

## Key Rules

- Always declare `pool` **globally** — avoids lifetime issues
- Pass `&pool` as the **last argument** to every pmr container constructor
- PMR containers are **drop-in replacements** — all methods work identically
- Mixing PMR and non-PMR containers in the same code is fine
- `unsynchronized_pool_resource` is your default choice in CP; switch to `monotonic_buffer_resource` only when you want max speed and no individual frees
